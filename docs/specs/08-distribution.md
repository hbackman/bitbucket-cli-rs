# 08 — Distribution

**Status:** Draft (Rust)
**Depends on:** [`01-architecture.md`](01-architecture.md)
**Slice goal:** A tagged release on GitHub produces working static binaries for every supported platform, a Homebrew tap is updated automatically, and `brew install hbackman/bb/bb` installs a runnable `bb`.

## Platforms

| OS | Arch | Target triple | Notes |
| --- | --- | --- | --- |
| macOS | x86_64 | `x86_64-apple-darwin` | Intel |
| macOS | aarch64 | `aarch64-apple-darwin` | Apple Silicon |
| Linux | x86_64 | `x86_64-unknown-linux-musl` | Static via musl |
| Linux | aarch64 | `aarch64-unknown-linux-musl` | |
| Windows | x86_64 | `x86_64-pc-windows-msvc` | Zipped; signing post-MVP |

Rust handles cross-compilation cleanly through `rustup target add`. There's no CGO equivalent — the `keyring` crate uses platform native APIs (Security framework on macOS, libsecret on Linux, Credential Manager on Windows) via Rust bindings, and reqwest uses `rustls-tls` (pure Rust) so no OpenSSL.

On Linux, the `keyring` crate depends on D-Bus / libsecret at runtime; for static `musl` builds we feature-flag it to `keyring = { version = "3", default-features = false, features = ["sync-secret-service", "vendored"] }` or fall back to the plaintext store (see spec 02). Decision pending until we have a working release build to test.

## Versioning

Semver. `0.x.y` until 0.1.0 ships; then 0.x.y until the surface stabilizes; then 1.0.0+.

Tagging convention: `v0.1.0`. Both `cargo-dist` and our hand-rolled workflow rely on git tags.

## Release tooling: `cargo-dist`

We use [`cargo-dist`](https://github.com/axodotdev/cargo-dist) — the de-facto Rust analog to GoReleaser. It generates the GitHub Actions release workflow, builds the matrix, builds the archives, and (optionally) updates the Homebrew tap.

`Cargo.toml` addition:

```toml
[workspace.metadata.dist]
cargo-dist-version = "0.21.0"
ci = ["github"]
installers = ["shell", "powershell", "homebrew"]
targets = [
  "x86_64-apple-darwin",
  "aarch64-apple-darwin",
  "x86_64-unknown-linux-musl",
  "aarch64-unknown-linux-musl",
  "x86_64-pc-windows-msvc",
]
tap = "hbackman/homebrew-bb"
publish-jobs = ["homebrew"]
pr-run-mode = "plan"
allow-dirty = ["ci"]

[workspace.metadata.dist.github-custom-runners]
aarch64-unknown-linux-musl = "ubuntu-24.04-arm"
```

After `cargo dist init`, two files appear:
- `.github/workflows/release.yml` — generated by cargo-dist
- `dist-workspace.toml` (or `Cargo.toml` metadata, as above)

We commit the generated workflow. Regenerate via `cargo dist generate` whenever the metadata changes.

### Embedded OAuth secrets

cargo-dist runs `cargo build --release` for each target. We thread the OAuth secrets through `build.rs` env vars (spec 02), so the GitHub Actions job needs them in scope:

```yaml
# Tweaks applied to the cargo-dist-generated release.yml
jobs:
  build-local-artifacts:
    env:
      BB_OAUTH_CLIENT_ID:     ${{ secrets.BB_OAUTH_CLIENT_ID }}
      BB_OAUTH_CLIENT_SECRET: ${{ secrets.BB_OAUTH_CLIENT_SECRET }}
```

`cargo-dist` supports `[workspace.metadata.dist.env]` or a `dist-extra.yml` patch — pick whichever the current cargo-dist version supports cleanly and document the choice in the PR that lands this slice.

## GitHub Actions

`.github/workflows/release.yml` is owned by `cargo-dist` (regenerate when bumping versions). The CI workflow from spec 01 (`.github/workflows/ci.yml`) stays as-is and runs on every PR:

- `cargo fmt --all -- --check`
- `cargo clippy --all-targets --all-features -- -D warnings`
- `cargo test --all`
- `cargo build --release`
- Matrix: `ubuntu-latest`, `macos-latest`, `windows-latest` × stable Rust.

(Spec 01 currently runs on ubuntu + macos only; adding Windows here.)

## Homebrew tap

Separate repo: `hbackman/homebrew-bb`. `cargo-dist` writes `Formula/bb.rb` on each release.

Install:
```
brew install hbackman/bb/bb
```

The tap repo needs:
- `README.md`
- `Formula/` directory (created by cargo-dist on first release)
- A fine-grained personal access token (`HOMEBREW_TAP_TOKEN` secret) with write access to that repo only.

## Code signing (post-MVP)

- **macOS notarization**: requires an Apple Developer account and `notarytool` integration. cargo-dist supports this when `apple-codesign` is configured; defer until we have user reports about Gatekeeper blocking.
- **Windows**: same — defer until a user complains.

Until then, the README documents `xattr -d com.apple.quarantine /usr/local/bin/bb` for users who download tarballs directly on macOS.

## Where the OAuth client secrets live

GitHub Actions secrets:
- `BB_OAUTH_CLIENT_ID`
- `BB_OAUTH_CLIENT_SECRET`
- `HOMEBREW_TAP_TOKEN`
- `GITHUB_TOKEN` (auto-provided)

The OAuth consumer is registered in a Bitbucket workspace owned by the project. Document who has admin access; rotate annually or on suspected exposure.

Local dev builds without these secrets work fine — `bb auth login` then requires `--client-id` / `--client-secret` (spec 02).

## Concrete deliverables

1. `[workspace.metadata.dist]` block in `Cargo.toml` with the contents above.
2. `cargo dist init && cargo dist generate` — committed `.github/workflows/release.yml`.
3. Patched env-var injection for `BB_OAUTH_CLIENT_ID` / `BB_OAUTH_CLIENT_SECRET`.
4. Updated `.github/workflows/ci.yml` to include Windows (matrix add).
5. Create the `hbackman/homebrew-bb` tap repo (manual, one-time).
6. Add the four GitHub Actions secrets: `BB_OAUTH_CLIENT_ID`, `BB_OAUTH_CLIENT_SECRET`, `HOMEBREW_TAP_TOKEN`, `GITHUB_TOKEN` (auto).
7. Register the OAuth consumer at https://bitbucket.org/account/settings/oauth-consumers (manual, one-time).
8. README install section.
9. A dry-run snapshot release via `cargo dist plan` and then `cargo dist build --tag v0.0.0-snapshot` to verify the matrix builds cleanly *before* tagging 0.1.0.

## Acceptance criteria

- `git tag v0.1.0 && git push --tags` triggers a green release workflow.
- The release page on GitHub shows tarballs/zips for each platform/arch, plus `sha256.txt` (cargo-dist's checksums file).
- `brew install hbackman/bb/bb` works on macOS Intel + Apple Silicon.
- `bb --version` from the installed binary prints `0.1.0` with a real commit SHA and date.
- `bb auth login` works against bitbucket.org using the embedded OAuth consumer.
- The Homebrew formula PR/commit is created automatically by cargo-dist on release.

## Open questions

- Whether to use cargo-dist or roll a hand-built release workflow with `actions/cross` + `softprops/action-gh-release`. **Lean: cargo-dist** — the generated workflow is mature and the alternative reinvents a lot of plumbing.
- Whether to keep an `unstable` Homebrew tap for nightly builds. **Lean: no** until we have multiple contributors.
- Whether to sign macOS binaries before 1.0. **Lean: no** — link to `xattr -d com.apple.quarantine` in the README.
- Whether to set up SLSA provenance / sigstore attestations (cargo-dist supports both via `--sign-with`). **Lean: defer to 0.2.**
- Linux keyring: ship libsecret-linked binary (dynamic libsecret) or pure-musl with plaintext fallback only? **Lean: ship both** — release one statically linked binary with plaintext storage and a separate `linux-gnu` build that links libsecret. Revisit after first release feedback.
